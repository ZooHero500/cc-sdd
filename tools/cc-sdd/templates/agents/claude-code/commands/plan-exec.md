# Plan Execution Workflow

<background_information>
- **Mission**: Execute an implementation plan generated by `/yy:feature`, following it step-by-step with TDD and code review
- **Success Criteria**:
  - Plan loaded and understood
  - Tasks executed in order with TDD
  - Checkpoints verified at each stage
  - Code review passed
  - Spec and changelog updated
</background_information>

<instructions>
## Core Task
Execute the implementation plan for spec **$1**. If no spec name provided, detect the most recent planned spec.

## Execution Steps

### Step 1: Load Context
1. If `$1` provided: Load `{{KIRO_DIR}}/specs/$1/`
2. If not provided: Search `{{KIRO_DIR}}/specs/` for specs with `status: "planned"`, pick the most recent
3. Read all spec files: `spec.json`, `requirements.md`, `design.md`, plan file(s)
4. Load entire `{{KIRO_DIR}}/steering/` directory as project context
5. Verify spec status is "planned" — if not, report current status and stop

### Step 2: Execute Plan
- Invoke the `superpowers:executing-plans` skill if available; otherwise follow the plan step-by-step manually
- Follow the plan step-by-step:
  - Implement each task using TDD (RED → GREEN → REFACTOR)
  - Run tests after each task
  - Checkpoint: verify passing tests before moving to next task
- If a step fails or is blocked:
  - Document the issue
  - Attempt to resolve
  - If unresolvable, stop and report to user

### Step 3: Auto-Complete
1. **Code Review**: Invoke the `superpowers:requesting-code-review` skill if available; otherwise perform manual review of all changes
   - If issues found → fix → re-review → until clean
2. **Update changelog**: Append entry with date, feature description, and affected files
3. **Update spec.json**: Set `status: "closed"`, update `updated_at`

### Step 4: Output Summary
- Tasks completed
- Files changed
- Tests added/modified
- Any issues encountered

## Critical Constraints
- **Follow the Plan**: Execute tasks in the order specified
- **TDD**: Write tests before implementation for each task
- **Checkpoint**: Verify tests pass between tasks
- **No Scope Creep**: Only implement what's in the plan
- **Report Blockers**: Don't silently skip failed steps
</instructions>

## Tool Guidance
- **Read**: Load all spec files before starting
- **Bash**: Run tests at each checkpoint
- **Grep/Glob**: Find relevant code referenced in the plan

## Output Description
Provide output in the language specified in spec.json:
1. **Plan Executed**: Feature name and total tasks
2. **Tasks Completed**: Count and brief per-task summary
3. **Files Changed**: Bullet list
4. **Tests**: New/modified tests
5. **Issues**: Any blockers encountered (if any)
6. **Status**: Spec closed

**Format**: Concise Markdown, under 300 words.

## Safety & Fallback

**Spec Not Found**:
- List available specs in `{{KIRO_DIR}}/specs/`
- Suggest: "Run `/yy:feature <description>` to create a new feature spec."

**Spec Not in Planned State**:
- Report current status
- If "open": "This spec needs a plan first. Run `/yy:feature` to generate one."
- If "closed": "This spec is already completed."

**Plan Step Fails**:
- Document failure in spec directory
- Report to user with context
- Do not proceed to next step until resolved
